Top productivity blockers — vote on suggested action points
10 / 16 respondents #1 blocker
Context switching — Google Chat, ad-hoc requests, support ops
The single most-cited blocker. Constant interruptions from chat messages, unplanned ad-hoc requests and support escalations fragment focus and make deep technical work very hard to sustain throughout the day.
"Context switching (Google Chat, ad-hoc requests, support/ops)" — picked by 10 of 16 people as a top-3 blocker
Suggested action points — vote for the ones you'd back
Protect a morning focus block — agree on a no-interrupt window (e.g. 9 am – 12 pm) on set days
Async-first norms for Google Chat — define "urgent" vs. "non-urgent" and set expected response SLAs per category
Consolidate ad-hoc support requests into a weekly triage slot — one predictable touchpoint instead of constant interruptions
Use status indicators consistently — e.g. "In deep work · back at 12 pm" to reduce unnecessary pings from teammates
8 / 16 respondents #2 blocker
Unclear requirements, missing specs, shifting priorities
Half the team cited vague or changing requirements as a top blocker. Work starts without clear acceptance criteria, leading to rework and misalignment. Broader product direction is also visible to only a small group, making it hard to understand the bigger picture.
"A lot of future product direction is still known only by a few people — with heavy cross-team communication, it's hard to see where the product is heading."
Suggested action points
Require acceptance criteria for every ticket before it enters a sprint — "no AC = no start" as a team norm
Introduce a lightweight PRD template for all features — problem statement, goal, constraints, non-goals, success metrics
Dedicated pre-sprint grooming session to surface and resolve ambiguities before any development begins
Quarterly roadmap review open to the whole engineering team — share upcoming priorities and the reasoning behind them
7 / 16 respondents Documentation
Hunting for information — fragmented docs, unclear ownership
7 people lose time searching for information scattered across GitBook, Webflow, Medium and internal channels. It's often unclear who owns what, leading to unnecessary colleague interruptions. Documentation satisfaction is 2.6 / 5 — joint lowest in the survey.
"I would document all internal knowledge, know-hows, processes, alignments — so I would not need to annoy/wait on people. And it would be usable for LLMs for context and better outcomes."
Suggested action points
Consolidate all docs into one source of truth — retire the scattered GitBook / Webflow / Medium setup in favour of a single system
Create and maintain an ownership map — one place to look up who owns which component, service or codebase area
Make doc updates part of the Definition of Done — every PR that changes behaviour must include a documentation update
Quarterly doc hygiene sessions — review, update and retire stale pages together as a team activity
3 people: 3+ day waits PR pipeline
PR review delays — no SLA, large PRs, inconsistent coverage
Three respondents experience 3+ day waits as a recurring bottleneck. Large PRs are hard to review quickly. The flip side is also frustrating: 10+ small trivial changes getting held up for formal approval. No defined SLA or rotation means coverage is unpredictable.
"One of my latest PRs took 4+ working days to get reviewed — requires a more systematic approach." / "When I have 10+ small changes unlikely to need comments, waiting for approvals becomes frustrating."
Suggested action points
Define a PR review SLA — e.g. first response within 1 business day, merge decision within 2 days
Introduce a recommended max PR size — e.g. ~400 LOC for standard PRs to keep review times reasonable
Set up a lightweight review rotation — always clear who is on review duty each week, no guessing
Fast-track path for low-risk small PRs — auto-merge after 1 approval if under 50 LOC with no review flags
Satisfaction 2.6 / 5 Δ1.4 — largest gap
Testing quality — the largest satisfaction gap in the survey
Testing is rated highly important (4.1 / 5) yet has the lowest satisfaction (2.6 / 5) — a gap of 1.43 points, the widest in the survey. Nearly half the team listed automated test coverage as their single "magic wand" wish. Manual testing effort is a significant drag on PR cycle times.
"We would have automated tests for everything." / "The testing was quite time-consuming — but it was required to make sure there are no false positives in the detection checks."
Suggested action points
Add minimum unit test coverage to the Definition of Done — new code cannot be merged without accompanying tests
Allocate dedicated sprint capacity for test infrastructure — one sprint per quarter focused purely on test tooling and coverage
Acquire Qase licence and integrate test management into the CI/CD pipeline for the portal (raised directly by QA)
Define a test pyramid policy — agree on target ratios for unit / integration / end-to-end tests per project type
